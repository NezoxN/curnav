# Функціональні вимоги до системи (System Functional Requirements)

Цей документ містить вичерпний перелік усіх функціональних вимог до системи управління навчальним процесом та формування індивідуальних освітніх траєкторій (ІОТ), заснований на архітектурі бази даних та реалізованих модулях.

## 1. Ролі користувачів та Модуль Аутентифікації
Система підтримує дві основні ролі з різними рівнями доступу: **Адміністратор (Admin)** та **Студент (Student)**.

- **FR1.1 Авторизація:** Користувачі повинні мати можливість увійти в систему використовуючи унікальну електронну пошту та пароль.
- **FR1.2 Управління доступом:** Система повинна обмежувати доступ до певних маршрутів (routes) та сторінок (pages) в залежності від ролі користувача.
- **FR1.3 Блокування акаунтів:** Система повинна дозволяти адміністраторам блокувати та розблоковувати акаунти користувачів (`isBlocked`). Заблоковані користувачі не можуть увійти в систему.
- **FR1.4 Безпека паролів:** Паролі повинні зберігатися у вигляді захищених хешів (`passwordHash`), система не повинна зберігати паролі у відкритому вигляді.

## 2. Адміністративна панель (Admin Dashboard)

### 2.1 Управління студентами (Student Management)
- **FR2.1.1 CRUD Студентів:** Адміністратор повинен мати можливість створювати, переглядати, редагувати та видаляти профілі студентів.
- **FR2.1.2 Прив'язка до академічної структури:** Під час створення або редагування студента, система повинна дозволяти вибрати його Освітню програму (Educational Program) та навчальну Групу (Group).
- **FR2.1.3 Атрибути студента:** Система повинна зберігати та дозволяти редагувати такі дані: ПІБ (`fullName`), поточний курс навчання (`currentYear`), форму навчання (`educationForm`).
- **FR2.1.4 Імпорт студентів:** Адміністратор повинен мати інструмент для масового імпорту даних студентів з файлів (CSV/Excel). Система повинна валідувати вхідні дані під час імпорту.

### 2.2 Управління групами (Group Management)
- **FR2.2.1 CRUD Груп:** Адміністратор повинен мати можливість створювати, редагувати та видаляти навчальні групи.
- **FR2.2.2 Прив'язка груп:** Кожна група повинна бути жорстко прив'язана до певної Освітньої програми.
- **FR2.2.3 Перегляд складу:** Адміністратор повинен мати можливість бачити список студентів, що належать до певної групи.

### 2.3 Управління дисциплінами (Course Management)
- **FR2.3.1 CRUD Дисциплін:** Адміністратор повинен мати можливість додавати нові дисципліни, редагувати їх властивості та видаляти їх з бази.
- **FR2.3.2 Атрибути дисципліни:** Система повинна підтримувати налаштування таких полів: назва, опис, кількість кредитів ECTS, тип контролю (іспит, залік тощо), семестр викладання та категорія.
- **FR2.3.3 Вибіркові дисципліни (Electives):** Адміністратор повинен мати можливість позначати дисципліни як вибіркові (прапорець `isSelective`).
- **FR2.3.4 Ліміти на зарахування:** Для вибіркових дисциплін адміністратор повинен мати можливість встановлювати максимальну кількість студентів, що можуть на них записатися (`maxStudents`).
- **FR2.3.5 Прив'язка до ОП:** Дисципліна може бути прив'язана до однієї або декількох Освітніх програм (`CourseEducationalProgram`), що визначає, яким студентам вона буде доступна.
- **FR2.3.6 Пререквізити (Course Dependencies):** Система повинна дозволяти створювати зв'язки між дисциплінами (батьківська -> дочірня) із зазначенням ваги зв'язку (`weight`). Це визначає, які курси є базовими для інших.
- **FR2.3.7 Імпорт дисциплін:** Система повинна мати функціонал для розширеного імпорту дисциплін з Excel/CSV, що включає парсинг пререквізитів та автоматичне зв'язування з Освітніми програмами під час завантаження.

### 2.4 Управління Освітніми програмами (Educational Program Management)
- **FR2.4.1 CRUD ОП:** Адміністратор повинен мати можливість створювати, редагувати та видаляти Освітні програми.
- **FR2.4.2 Фінансово-академічні ліміти:** Система повинна дозволяти встановлювати глобальні параметри для ОП: загальну кількість кредитів за весь період (`totalCredits`) та максимальний дозволений ліміт кредитів на один семестр (`maxCreditsPerSem`). Це правило використовуватиметься для валідації вибору студента.

### 2.5 Управління успішністю (Grade Management)
- **FR2.5.1 Академічні записи:** Адміністратор повинен мати можливість фіксувати результати навчання студентів (створювати `AcademicRecord`).
- **FR2.5.2 Деталі оцінювання:** Кожен запис повинен містити студента, дисципліну, назву контрольного заходу (`assessmentName`), отриманий бал (`gradeValue`) та семестр проходження.
- **FR2.5.3 Імпорт оцінок:** Адміністратор повинен мати можливість масового імпорту відомостей з оцінками з файлу.

### 2.6 Рецензування індивідуальних траєкторій (Trajectory Review)
- **FR2.6.1 Перегляд заявок:** Адміністратор повинен мати доступ до списку всіх поданих студентами індивідуальних освітніх траєкторій (ІОТ).
- **FR2.6.2 Управління статусами:** Адміністратор може затверджувати або відхиляти заявки. Відхилені заявки повинні повертатися студенту на доопрацювання.

### 2.7 Адміністрування системи (System Administration)
- **FR2.7.1 Керування кампанією вибору:** В налаштуваннях (`GlobalSettings`) адміністратор має вмикати та вимикати можливість формування траєкторій для студентів (тумблер `isSelectionOpen`).
- **FR2.7.2 Управління адміністраторами:** Головний адміністратор може створювати нові акаунти адміністраторів та керувати існуючими.
- **FR2.7.3 Керування категоріями:** Можливість створювати категорії (`CourseCategory`) для логічного групування дисциплін.

## 3. Портал Студента (Student Portal)

### 3.1 Формування індивідуальної освітньої траєкторії (Trajectory Selection)
- **FR3.1.1 Доступність модуля:** Модуль вибору дисциплін повинен бути активним лише тоді, коли глобальне налаштування `isSelectionOpen` встановлено в `true`.
- **FR3.1.2 Каталог вибіркових дисциплін:** Студент повинен бачити список доступних вибіркових дисциплін, що прив'язані до його Освітньої програми.
- **FR3.1.3 Відображення наповненості:** Для кожної дисципліни з лімітом місць повинно відображатися кількість вже записаних студентів та загальний ліміт (`maxStudents`).
- **FR3.1.4 Валідація ліміту місць:** Система не повинна дозволяти студенту обрати дисципліну, якщо кількість заявок на неї вже досягла максимуму.
- **FR3.1.5 Валідація кредитів:** Система повинна рахувати сумарну кількість кредитів ECTS обраних дисциплін та забороняти подачу заявки, якщо сума перевищує `maxCreditsPerSem` для Освітньої програми студента.
- **FR3.1.6 Подача заявки:** Студент повинен мати можливість сформувати список дисциплін (`TrajectoryItems`) та відправити траєкторію на затвердження адміністрації.

### 3.2 Профіль студента та Успішність (Student Profile & Records)
- **FR3.2.1 Перегляд профілю:** Студент може переглядати свою персональну інформацію, поточний курс, групу та освітню програму.
- **FR3.2.2 Електронна заліковка:** Студент має доступ до перегляду всіх своїх академічних записів (оцінок) у розрізі семестрів та дисциплін.
- **FR3.2.3 Статус траєкторії:** Студент повинен бачити поточний статус своєї заявки на ІОТ (На розгляді, Затверджено, Відхилено).

## 4. Аналітика та Моделювання (BKT - Bayesian Knowledge Tracing)
- **FR4.1 Параметри моделі:** Система повинна передбачати збереження математичних параметрів для моделювання знань студента з кожної дисципліни: `pLearn` (ймовірність засвоєння), `pSlip` (ймовірність випадкової помилки), `pGuess` (ймовірність вгадування), `currentPKnown` (поточний рівень знань).
- **FR4.2 Трекінг прогресу:** Ці параметри оновлюються на основі академічних досягнень (оцінок) та можуть використовуватися системою для алгоритмічної рекомендації пререквізитів або додаткових матеріалів.

## 5. Загальні системні вимоги
- **FR5.1 Цілісність даних:** Система повинна підтримувати каскадне видалення (Cascade Delete). Наприклад, видалення курсу автоматично видаляє його розклад, пререквізити та записи з траєкторій.
- **FR5.2 Уніфікований UI/UX:** Усі таблиці, форми, кнопки та індикатори завантаження повинні бути стандартизованими та використовувати єдину дизайн-систему (на базі Mantine).
- **FR5.3 Пошук та фільтрація:** В адміністративних панелях повинні бути реалізовані механізми пошуку (за назвою, ПІБ) та фільтрації (за категоріями, типами, освітніми програмами) для швидкої навігації великими масивами даних.
