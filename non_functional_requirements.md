# Нефункціональні вимоги до системи (System Non-Functional Requirements)

Цей документ містить деталізований перелік усіх нефункціональних вимог до системи управління навчальним процесом та формування індивідуальних освітніх траєкторій (ІОТ). Він описує атрибути якості, технічні обмеження, стандарти безпеки та архітектурні вимоги, необхідні для ефективного функціонування усіх програмних модулів системи.

## 1. Продуктивність та Швидкодія (Performance)
- **NFR1.1 Час відгуку інтерфейсу:** Час реакції системи на стандартні запити користувачів (навігація між сторінками, завантаження списків дисциплін, перегляд академічного профілю) не повинен перевищувати 1-2 секунд при нормальному мережевому з'єднанні.
- **NFR1.2 Швидкість масових операцій:** Процеси імпорту об'ємних масивів даних з Excel/CSV (до 5000 записів за раз: списки студентів, каталог дисциплін, результати оцінювання) повинні парситися, валідуватися та зберігатися в базу даних не довше ніж за 30-60 секунд.
- **NFR1.3 Високе навантаження (Конкурентність):** Архітектура системи повинна гарантувати стабільну роботу під час пікових навантажень — зокрема в періоди кампаній вибору ІОТ, коли в систему можуть одночасно увійти до 1000-2000 студентів для вибору дисциплін.
- **NFR1.4 Обчислювальна оптимізація:** Аналітичні алгоритми (зокрема BKT - Bayesian Knowledge Tracing для трекінгу знань) та механізми перевірки пререквізитів дисциплін повинні бути оптимізовані, щоб не блокувати головний потік виконання серверу (event loop).
- **NFR1.5 Оптимізація передачі даних (Пагінація):** Для таблиць із великими обсягами інформації (управління студентами, оцінками) система повинна реалізовувати серверну пагінацію (pagination) та ліниве завантаження (lazy loading), що мінімізує споживання оперативної пам'яті як на клієнті, так і на сервері.
- **NFR1.6 Кешування:** Система повинна використовувати механізми in-memory кешування для даних, що рідко змінюються (довідники, категорії курсів, структура освітніх програм), для зменшення кількості запитів до бази даних.

## 2. Безпека та Захист даних (Security)
- **NFR2.1 Хешування паролів:** Усі паролі користувачів системи категорично заборонено зберігати у відкритому вигляді. Система повинна використовувати криптографічно надійні алгоритми хешування (наприклад, bcrypt або Argon2) із застосуванням унікальної «солі» (salt).
- **NFR2.2 Захист API та Авторизація:** Доступ до всіх серверних ендпоінтів повинен бути захищений стандартом JWT (JSON Web Tokens). Сервер повинен валідувати токен та перевіряти роль користувача (Student/Admin) перед виконанням будь-якої дії (Role-Based Access Control).
- **NFR2.3 Захист від ін'єкцій:** Всі вхідні дані від клієнта (тіла запитів, параметри) повинні проходити сувору валідацію (через схеми Joi/Zod). Для доступу до бази даних PostgreSQL повинен застосовуватися ORM Prisma, який автоматично санітизує запити та запобігає SQL-ін'єкціям.
- **NFR2.4 Захист від типових вразливостей веб-застосунків:** Система має бути захищена від атак XSS (Cross-Site Scripting) та CSRF (Cross-Site Request Forgery) за рахунок вбудованих механізмів React/Next.js, правильного налаштування CORS політик та безпечних HTTP-заголовків (наприклад, Helmet.js).
- **NFR2.5 Управління сесіями:** Токени доступу (Access Tokens) повинні мати короткий термін дії (наприклад, 1-2 години). Безперервність сесії має забезпечуватися через безпечний механізм Refresh Tokens, які передаються через захищені httpOnly куки.

## 3. Надійність та Відмовостійкість (Reliability & Availability)
- **NFR3.1 Доступність (Uptime):** Очікуваний час безперебійної роботи серверної частини та бази даних становить не менше 99.9% протягом семестру та 99.99% під час критичних кампаній запису на вибіркові дисципліни.
- **NFR3.2 Транзакційна цілісність (ACID):** Будь-які комплексні операції запису (наприклад, подача студентом траєкторії, яка зменшує ліміти місць, видалення сутностей, або імпорт зв'язаних сутностей) повинні виконуватися в межах жорстких транзакцій бази даних. При виникненні помилки зміни повинні відкочуватися (rollback), не допускаючи часткового збереження даних.
- **NFR3.3 Обробка виняткових ситуацій (Error Handling):** Система не повинна припиняти роботу («падати») через непередбачувані помилки. Глобальні обробники винятків (Global Error Handler) повинні логувати технічні деталі проблеми, а клієнту повертати безпечну, стандартизовану HTTP-відповідь без розкриття стеку виконання (stack trace).
- **NFR3.4 Резервне копіювання (Backups):** База даних повинна підтримувати регулярне автоматизоване резервне копіювання для швидкого відновлення системи у випадку аварії або випадкової втрати інформації.

## 4. Зручність використання (Usability & UX/UI)
- **NFR4.1 Адаптивність (Responsive Design):** Інтерфейс користувача як на порталі студента, так і в адміністративній панелі, повинен автоматично адаптуватися до різних розмірів екранів (персональні комп'ютери, ноутбуки, планшети, мобільні пристрої, breakpoints: 320px, 768px, 1024px, 1440px).
- **NFR4.2 Кросбраузерність:** Система повинна коректно відображатися і повноцінно функціонувати в останніх версіях популярних веб-браузерів (Google Chrome, Safari, Mozilla Firefox, Microsoft Edge).
- **NFR4.3 Консистентність дизайну:** Інтерфейс повинен використовувати єдину дизайн-систему компонентів (на базі бібліотеки Mantine v7). Усі таблиці, форми, кнопки, модальні вікна та типографіка повинні мати стандартизований вигляд, що забезпечує професійний та сучасний (premium) UX. Передбачити підтримку Dark Mode / Light Mode.
- **NFR4.4 Зворотній зв'язок системи:** Кожна дія (збереження форми, помилка валідації, успішний імпорт) повинна супроводжуватися візуальними сповіщеннями (toast notifications). Довготривалі операції (завантаження даних, імпорт) обов'язково повинні супроводжуватись індикаторами процесу (loaders/skeletons).
- **NFR4.5 Локалізація:** Інтерфейс та всі повідомлення системи повинні бути написані виключно зрозумілою українською мовою з дотриманням діючої академічної термінології вищих навчальних закладів.

## 5. Підтримуваність та Архітектура (Maintainability)
- **NFR5.1 Технологічний стек:** Frontend: React 18+; Backend: Node.js (v18/v20) + Express/NestJS, Python (FastAPI/Flask) для аналітичного модуля; БД: PostgreSQL 15+. 
- **NFR5.2 Типізація та стандарти коду:** Увесь вихідний код на клієнті та сервері має бути написаний мовою TypeScript із застосуванням суворих налаштувань типізації (`strict` mode). Код має відповідати єдиним правилам форматування (ESLint/Prettier).
- **NFR5.3 Модульність та Шаблони проектування:** Бекенд повинен дотримуватися багатошарової архітектури (Controllers -> Services -> Repositories/ORM). Логіка роботи з базою даних повинна бути відділена від логіки обробки HTTP-запитів.
- **NFR5.4 Документування бази даних:** Архітектура бази даних повинна підтримуватись в актуальному стані у форматі DBML (Database Markup Language).
- **NFR5.5 Конфігурація середовища:** Усі критичні змінні, що залежать від середовища (Development, Production), такі як паролі БД, ключі, порти, повинні зберігатися у зовнішніх конфігураційних файлах (`.env`) і не потрапляти до репозиторію.
- **NFR5.6 Логування та Моніторинг:** Система повинна вести журнали (логи) ключових системних подій, транзакцій та помилок для можливості ефективного аудиту та відлагодження.

## 6. Сумісність та Інтеграційні можливості (Interoperability)
- **NFR6.1 Формати обміну даними:** Система повинна підтримувати обробку стандартних форматів `.csv` та `.xlsx` для імпорту контингенту студентів, навчальних планів та експорту успішності, що дозволяє легко інтегруватися з наявними в університеті системами документообігу (наприклад, ЄДЕБО).
- **NFR6.2 Стандартизація API:** Взаємодія між frontend і backend частинами повинна здійснюватися через RESTful API (JSON), що в майбутньому дозволить підключати до системи мобільні додатки або зовнішні аналітичні сервіси.
- **NFR6.3 Розгортання (Deployment):** Архітектура повинна підтримувати контейнеризацію за допомогою Docker, що спрощує процес розгортання в будь-якому хмарному середовищі (CI/CD pipeline).
